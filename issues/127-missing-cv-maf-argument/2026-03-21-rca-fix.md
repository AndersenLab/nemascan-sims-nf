## Root Cause Analysis

### Summary

`write_gwa_to_db.R:86` calls `generate_trait_id()` with 5 positional arguments, but
the function was promoted to a v=2 signature requiring 7. The two new parameters —
`cv_maf_effective` and `cv_ld` — encode causal-variant pool metadata in the trait hash
and have no defaults. The script's `option_list` was never extended to accept them, the
`main.nf` module never forwards them, and the upstream `ch_gwa_db_inputs` channel never
carries them. All three DB analysis scripts (`analyze_qtl.R`, `assess_sims.R`,
`write_trait_data.R`) received this update; `write_gwa_to_db.R` was missed.

### Diagnostic Steps

1. Ran `NXF_VER=24.10.4 nextflow run main.nf -profile test_cv_pool,docker`; pipeline
   failed in `DB_MIGRATION_WRITE_GWA_TO_DB`.
2. Inspected `.command.err` in the failed work directory — R halted with
   `argument "cv_maf_effective" is missing, with no default`.
3. Traced the call stack to `write_gwa_to_db.R:86`:
   `generate_trait_id(ms_id$hash, params$nqtl, params$effect, params$rep, params$h2)`
   — confirmed 5-arg call against the 7-arg signature in `R/database.R:479`.
4. Compared `write_gwa_to_db/main.nf` against `analyze_qtl/main.nf` — the latter
   carries `val(cv_maf_effective)` and `val(cv_ld)` in its input tuple and passes
   `--cv_maf_effective ${cv_maf_effective} --cv_ld ${cv_ld}` in the script block.
5. Audited `main.nf` — found that the G1 fork (lines 221-228) creates three consumers
   of `ch_sf.cv_maf_vals` (`for_plink`, `for_trait`, `for_analysis`), but the GWA write
   path has no corresponding fork. `ch_cv_maf_keyed_for_analysis` is a queue channel
   consumed by the analysis path and cannot be reused.

### Key Evidence

**`.command.err`:**

### Confirmed Root Cause

The v=2 `generate_trait_id()` signature was deployed to three of the four DB migration
scripts but not to `write_gwa_to_db.R`. The channel infrastructure to carry
`cv_maf_effective` to that process was also never added to `main.nf`.

## Solution

### Approach

Thread `cv_maf_effective` and `cv_ld` to `DB_MIGRATION_WRITE_GWA_TO_DB` by extending
the existing G1 fork pattern — adding a fourth `.tap{}` branch for the GWA write path,
combining it into `ch_gwa_db_inputs`, updating the module's input tuple, and fixing the
R script's `option_list` and `generate_trait_id()` call. This is the same pattern
already used by `analyze_qtl/main.nf` (params tuple carries both cv params).

Three files require changes.

### Changes Made

**`main.nf` — G1 fork: add fourth consumer branch (lines 221-228)**

```groovy
// before

    .tap { ch_cv_maf_for_plink }
    .tap { ch_cv_maf_for_trait }
    .map { meta, cv_maf_eff -> tuple(meta.id, cv_maf_eff) }
    .set { ch_cv_maf_keyed_for_analysis }

// after
ch_sf.cv_maf_vals
    .tap { ch_cv_maf_for_plink }
    .tap { ch_cv_maf_for_trait }
    .tap { ch_cv_maf_for_gwa_write }        // fourth consumer (write_gwa_to_db)

    .set { ch_cv_maf_keyed_for_analysis }

ch_cv_maf_keyed_for_gwa_write = ch_cv_maf_for_gwa_write
    .map { meta, cv_maf_eff -> tuple(meta.id, cv_maf_eff) }
```

### Rationale

The `.tap{}` / keyed-combine pattern is already established in this codebase for
threading per-group scalar values to downstream processes. Adding a fourth tap in G1
requires touching only the fork block, `ch_gwa_db_inputs` construction, the module
input, and the R script — no changes to the execution graph or barrier logic. Folding
both cv params into the main params tuple (rather than as separate `val` inputs)
matches the `analyze_qtl` module, which has an identical requirement.

## Options Considered

### Option A — Recommended: extend params tuple (chosen above)

Fold `cv_maf_effective` and `cv_ld` into `ch_gwa_db_inputs` via a new G1 fork tap +
keyed `.combine()`. This exactly mirrors the `DB_MIGRATION_ANALYZE_QTL` pattern, which
already extends its params tuple with both cv values.

**Pros:** Consistent with `analyze_qtl/main.nf`; single channel input to the process;
no structural change to the calling site in `main.nf`.
**Cons:** Params tuple grows to 14 elements (currently 12); slightly more verbose `map`
closure.

### Option B — Separate `val` inputs (write_trait_data pattern)

Pass `cv_maf_effective` as a separate channel and `cv_ld` as `Channel.value(cv_ld)`,
matching the `DB_MIGRATION_WRITE_TRAIT_DATA` call signature.

```groovy
// main.nf call site
DB_MIGRATION_WRITE_GWA_TO_DB(
    ch_gwa_db_inputs,            // 12-element tuple unchanged
    GCTA_PERFORM_GWA.out.gwa,
    db_output_dir,
    ch_cv_maf_for_gwa,           // separate val channel
    Channel.value(cv_ld)         // scalar
)
```

This requires `multiMap`-splitting the combined channel to separate `cv_maf` from the
params tuple, which adds ~5 lines of orchestration. The `ch_cv_maf_keyed_for_gwa`
channel must still be forked from G1 — no reduction in the number of touch points.
**Pros:** Cleaner process input signature; keeps the params tuple at 12 elements.
**Cons:** Creates a second convention for passing cv params (inconsistent with
`analyze_qtl`); requires additional `multiMap` in `main.nf`; two separate channel
inputs must stay in sync.

### Option C — Read `cv_maf_effective`/`cv_ld` from the database

Since `write_trait_data.R` writes both values to `trait_metadata.parquet` before
`write_gwa_to_db.R` runs, the script could query the DB at startup to look them up.

**Rejected:** `write_trait_data` and `write_gwa_to_db` run concurrently — there is no
guarantee that the trait record exists at the time the GWA writer starts. Adding an
explicit dependency edge (e.g., gating on `write_trait_data.out.done`) would serialize
what is currently parallel work and reduce throughput in large runs.
