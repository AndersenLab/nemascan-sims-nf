Analysis

Error classification

This is a different failure mode from issue #143. Issue #143 was "the Vp is below 1e-5" (phenotype variance too small for GCTA’s internal threshold, fixed by iterative Vp scaling in GCTA_MAKE_GRM). The current error is "fastGWA-REML can't converge".

Root cause: Vg > Vp (h² > 1)

The GCTA log reveals the underlying condition:

Best guess Vg range: 115.647573..., Vp: 115.362665...

Vg (115.648) > Vp (115.363) — the REML grid search converges numerically (step size reaches ~6e-11 by iteration 13) but the MLE solution has estimated genetic variance exceeding total phenotypic variance, implying h² ≈ 1.0025. Since h² must lie in [0, 1], the solution is outside the valid parameter space. GCTA detects this and hard-fails rather than clamping to the boundary (h² = 1).

Mechanism: sparse GRM distortion

The failure chain is:

  1. GCTA_MAKE_GRM builds the full dense GRM and runs an iterative REML loop, scaling phenotype until full-GRM Vp ≥ 1e-4. This passes — Vp = 115.363 is large.
  2. GCTA_PERFORM_GWA (inbred mode) sparsifies the GRM with --make-bK-sparse 0.05, zeroing all relatedness entries below 0.05.
  3. --fastGWA-mlm-exact then estimates Vg against the sparse GRM structure. For small, highly inbred C. elegans populations (n=100), sparsification changes the eigenstructure enough that the sparse-GRM MLE for Vg exceeds Vp.
  4. GCTA reports “can’t converge” because h² > 1.

Parameters in the failing task confirm the at-risk profile: nqtl=5 rep=32 h2=0.8 inbred nopca ce.n100.r06 maf=0.05 — small sample size (n=100), high simulated heritability (0.8), inbred mode.

Why the existing fix and planned safety net do not cover this

Issue #143 fix Planned safety net (future-gwa-safety-net.md) This bug
Error message "the Vp is below 1e-5" "the Vp is below 1e-5" (grep target) "fastGWA-REML can't converge"
Condition Vp < 1e-5 (absolute) Vp < 1e-5 (absolute) Vg > Vp (ratio, h² > 1)
Fix approach Scale pheno ×1000 Scale pheno ×1000 + retry Scaling is ineffective — the ratio Vg/Vp is scale-invariant

Specifically: the grep in the planned safety net (grep -q "the Vp is below 1e-5") would not match "fastGWA-REML can't converge", so the else branch would fire and immediately exit \${GWA_EXIT} — no benefit.

And even if the grep were extended to match this error, scaling the phenotype would not help: multiplying all phenotype values by k scales both Vg and Vp by k², leaving h² = Vg/Vp unchanged.

Retry behavior

The execution summary shows failed: 4, retries: 4 for GCTA_MAKE_GRM and failed: 5, retries: 4 for GCTA_PERFORM_GWA. The rockfish.config maxRetries = 3 setting was designed for transient SLURM failures. This GCTA convergence failure is deterministic — the same phenotype with the same sparse GRM will always fail. All retries are wasted compute.

Proposed fix direction

The fix must handle the case where --fastGWA-mlm-exact with --grm-sparse converges to h² > 1. Two options:
