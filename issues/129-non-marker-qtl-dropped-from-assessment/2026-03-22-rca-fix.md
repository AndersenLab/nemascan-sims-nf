## Root Cause Analysis

### Summary

`score_causal_markers()` (`R/assessment.R:170`) filters out causal variants with NA
`log10p`. Non-marker causal variants (positions outside the GWA marker set) have no
entry in the GWA output and therefore receive NA `log10p` after the left-join with
mapping data. The filter drops them before they reach `build_assessment_union()`, so
they never appear in the assessment TSV — not as FN rows, not with `Simulated.QTL.VarExp`
populated.

### Diagnostic Steps

1. Ran the HPC verification script from `testing_plan.qmd` against
   `results_hpc_cv_pool/`. Script failed at `stopifnot(nrow(non_marker_with_varexp) > 0)`
   with the message `Non-marker QTL with non-NA var.exp: 0`.

2. Confirmed `non_marker_qtl` is non-empty (the earlier `stopifnot(length(...) > 0)`
   passed), so the issue is that non-marker QTLs are absent from `res` entirely, not
   merely missing their var.exp.

3. Inspected all 20 rows of `results_hpc_cv_pool/db_simulation_assessment_results.tsv`:
   QTL column contains only `3:6026848` (the one marker-position causal variant),
   `2:9647114`, and `2:2511725` (detected FP peaks). The 4 non-marker causal variants
   (`1:2217428`, `2:9853034`, `2:11624318`, `3:16593852`) do not appear in any row.

4. Traced the DB path: `assess_sims.R` reads causal genotypes, calls
   `compute_var_exp_anova()` with the merged genotype matrix, augments `causal_variants`
   with `Simulated.QTL.VarExp`, then calls `compile_full_assessment()` →
   `score_causal_markers()`. Inspected `score_causal_markers()`:

   ```r

     dplyr::mutate(QTL = marker) %>%
     dplyr::filter(QTL %in% causal_variants$QTL) %>%   # only GWA positions match
     dplyr::select(QTL, log10p, significant) %>%
     dplyr::filter(!duplicated(QTL))

   effects_scores <- causal_variants %>%
     dplyr::left_join(causal_marker_scores, by = "QTL") %>%
     dplyr::filter(!is.na(log10p))                      # ← drops non-marker variants
   ```

   Non-marker causal variants are not in `mapping_data` (GWA output contains only
   marker positions), so the left-join gives them NA `log10p`, and the filter removes
   them.

5. Confirmed the local Docker Assertion 3 passed for a different reason: it checks only
   `sum(!is.na(Simulated.QTL.VarExp)) > 0` — satisfied by the single marker-position
   causal variant that survives the filter. The HPC check is strictly stronger.

6. Confirmed per-trait causal genotype Parquet is correctly written (1000 rows in
   `results_hpc_cv_pool/db/traits/causal_genotypes/147b00054e04d5e7402e_causal_geno.parquet`).
   The ANOVA computation itself is not the failure point; the results are discarded
   downstream.

### Confirmed Root Cause

`score_causal_markers()` (`R/assessment.R:170`) drops any causal variant whose position
is absent from the GWA output via `filter(!is.na(log10p))`. This is correct for the
marker-only case but incorrectly excludes non-marker causal variants in CV pool runs,
where causal positions outside the GWA marker set are intentional and should appear as
FN rows with `Simulated.QTL.VarExp` populated.

---

## Solution

### Approach

Remove `dplyr::filter(!is.na(log10p))` from `score_causal_markers()`. Non-marker causal
variants will then remain in `effects_scores` with NA `log10p` and NA `significant`.
`build_assessment_union()` already derives the `Simulated` and `Detected` flags by set
membership:

```r
Simulated = (QTL %in% effects_scores$QTL)   # TRUE for non-marker variants
Detected  = (QTL %in% overlap_df$QTL)       # FALSE — no GWA peak can match a non-marker
```

Their `Simulated.QTL.VarExp` (augmented onto `causal_variants` before
`compile_full_assessment()` is called) flows through the left-join unchanged.
`format_assessment_tsv()` and `designate_qtl()` both handle NA `log10p` already.

No other files require changes.

### Changes Made

**`R/assessment.R` — remove NA log10p filter from `score_causal_markers()`**

```r
# before (lines 167–172)
effects_scores <- causal_variants %>%
  dplyr::left_join(causal_marker_scores, by = "QTL") %>%
  dplyr::filter(!is.na(log10p))

# after
effects_scores <- causal_variants %>%
  dplyr::left_join(causal_marker_scores, by = "QTL")
```

### Rationale

The filter was written for the marker-only case where every causal variant has a GWA
score. Removing it lets non-marker causal variants appear as FN rows (`Simulated=TRUE,
Detected=FALSE`) with `log10p=NA`, which is correct: they were simulated but GCTA never
tested them because they are not in the marker set. Their `Simulated.QTL.VarExp` is
already computed correctly from the per-trait causal genotype Parquet by
`compute_var_exp_anova()`.

## Concordance Impact

The fix does **not** break the concordance tests between the DB and legacy assessment
paths. Three reasons:

1. **The integration data is collected from the standard `test` profile** (where
   `cv_maf = ms_maf` and all causal variants are marker positions). The fix only affects
   behavior for non-marker causal variants, which do not arise in that context.

2. **The legacy path applies the same filter.** `bin/Assess_Sims.R:121` also drops rows
   with `!is.na(log10p)`, so non-marker causal variants are absent from the legacy TSV.
   The concordance test uses `inner_join` on `QTL`: rows present only in the DB path
   (the new FN rows for non-marker positions) are excluded from the join and do not
   affect concordance scores.

3. **The `Simulated.QTL.VarExp` concordance test** (`test-assessment_cross_validation.R:414`)
   also uses `inner_join` on QTL followed by `filter(!is.na(varexp_legacy) & !is.na(varexp_db))`.
   Non-marker rows are absent from legacy, so they are excluded from both the join and
   the filter.

The fix introduces a by-design structural difference between the two paths for CV pool
runs: non-marker causal variants appear in the DB assessment as FN rows with var.exp
populated; they are absent from the legacy output entirely. This is the intended
behaviour — the CV pool feature is DB-path only. A comment should be added to
`test-assessment_cross_validation.R` near the existing design-difference note
(line 397) to document that non-marker FN rows are excluded from concordance by the
inner join, not by accident.
